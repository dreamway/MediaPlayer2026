# WZMediaPlayer 重构计划

基于 QMPlayer2 的成熟架构，对现有 WZMediaPlayer 进行分阶段重构，以提高代码可读性、可维护性和稳定性。

## 总体目标

参考 QMPlayer2 的多线程架构，将现有耦合度高的代码拆分为清晰的模块：
- **PlayClass**: 主控制器，协调所有线程
- **DemuxerThr**: 解复用线程
- **VideoThr/AudioThr**: 解码和播放线程
- **PacketBuffer**: 线程安全缓冲区
- **Decoder**: 解码器接口
- **VideoWriter**: 渲染器接口

## 重构原则

1. **渐进式重构**：分阶段进行，确保每个阶段都能编译出可执行程序进行功能验证
2. **保持兼容性**：重构过程中保持对外接口不变，用户体验不受影响
3. **单元测试**：每个模块重构后进行充分测试
4. **文档同步**：更新相关文档和注释

## 当前架构问题分析

### 现有架构
```
Movie (解复用+控制)
├── Video (视频解码)
├── Audio (音频解码)
└── FFmpegView (渲染)
```

### 主要问题
1. **职责混乱**：Movie 类承担解复用、解码控制、播放控制等多种职责
2. **线程管理混乱**：线程创建和管理逻辑分散
3. **同步机制缺失**：缺乏以音频时钟为基准的音视频同步
4. **接口耦合**：渲染和解码逻辑紧耦合
5. **错误处理不完善**：缺乏健壮的错误恢复机制
6. **缓冲管理简单**：缓冲区控制策略较为基础

### 评估结论

#### 架构对比分析

**参考代码架构（reference/qtvideoplayer）**：
- `AudioItem`：音频输出层，使用 Qt 的 `QAudioSink`
- `AudioThread`：音频解码线程，解码+重采样，通过信号槽与 AudioItem 通信

**当前 WZMediaPlayer 架构**：
- `Audio`：音频输出层，使用 OpenAL（类似 AudioItem）
- `AudioThread`：音频解码线程，解码+重采样，直接调用 Audio::writeAudio()（类似 AudioThread）

**对比结果**：
- ✅ **架构设计理念一致**：职责分离清晰（Audio 负责输出，AudioThread 负责解码）
- ✅ **功能等价**：当前实现与参考代码在功能上等价
- ⚠️ **主要差异是技术选型**：OpenAL vs QAudioSink，直接调用 vs 信号槽
- ⚠️ **需要优化的是稳定性**：错误处理、线程安全、资源管理、异常恢复
