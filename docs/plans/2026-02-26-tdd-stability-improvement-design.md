# WZMediaPlayer TDD 稳定性改进设计

> **日期**: 2026-02-26  
> **目标**: 使用 TDD 方式提升 3D 播放器稳定性，维持现有功能  
> **状态**: 待用户批准

## 一、背景与问题

### 1.1 当前状态

- **架构**: 已完成 VideoRenderer、AVClock、StereoOpenGLRenderer 重构
- **测试**: pywinauto 闭环测试框架已就绪（basic、seek、audio、sync、3d、edge、fullscreen、hw）
- **BUG**: 多个关键 BUG 重复修改多次，验证状态多为「待确认」

### 1.2 核心问题

1. **修复未经验证**: BUG-001/002/003/004/005 等修复已实施，但未通过自动化测试验证
2. **缺乏回归保护**: 修复后无自动化测试保障，易再次引入同类问题
3. **根因分析不足**: 部分 BUG 多次修复仍复发，需系统性调试

### 1.3 用户目标

- 使用 **Superpowers TDD** 技能：先写失败测试，再实现修复
- 提升 **稳定性**，维持 **现有功能**
- 采用 **规划优先**，再持续重构

## 二、方案对比

### 方案 A：先修复后验证（不推荐）

- **流程**: 直接修改代码 → 手动/自动化测试
- **问题**: 与以往做法相同，易陷入「改完又坏」循环
- **风险**: 高

### 方案 B：纯 TDD 逐 BUG 修复（可行）

- **流程**: 为每个 BUG 写最小失败测试 → 实现修复 → 验证通过
- **优点**: 每个修复有测试保障
- **缺点**: 需先确认当前哪些测试会失败，否则可能「测试本来就过」

### 方案 C：验证优先 + TDD（推荐）✅

- **流程**:
  1. **建立基线**: 编译项目，运行完整测试套件，记录通过/失败
  2. **按优先级**: 从 P0 到 P2 逐个 BUG 处理
  3. **每 BUG 的 TDD 循环**:
     - RED: 编写/确认能复现该 BUG 的最小失败测试
     - 运行测试，确认失败原因符合预期
     - GREEN: 实现最小修复
     - 运行测试，确认通过
     - REFACTOR: 清理代码，保持测试通过
  4. **系统性调试**: 若 3 次修复仍失败，按 systematic-debugging 技能质疑架构

- **优点**:
  - 基线明确，避免「误判修复」
  - 每个修复有回归测试
  - 与 systematic-debugging 结合，减少盲目修改

## 三、推荐方案详细设计（方案 C）

### 3.1 架构原则

- **测试优先**: 无失败测试不写生产代码
- **最小修复**: 每次只解决一个根因，不捆绑重构
- **进程监控**: 利用现有 `process_output_monitor` 捕获 QMutex、Access violation、ALSOFT 等

### 3.2 BUG 优先级与测试映射

| 优先级 | BUG | 测试类别 | 验证方式 |
|--------|-----|----------|----------|
| P0 | BUG-001 播放/切换崩溃 | basic | 进程存活、无 Access violation |
| P0 | BUG-002 音视频同步 | sync | 长时间播放、Seek 后播放 |
| P1 | BUG-003 Seek 后无声音 | seek, sync | Seek 后视频继续播放、无 ALSOFT 超阈值 |
| P1 | BUG-004 进度条不同步 | seek | 时间标签与 Seek 一致 |
| P1 | BUG-005 黑屏闪烁 | basic, edge | 图像验证非黑屏 |
| P2 | BUG-006 硬解码黑屏 | hw | 配置/日志检查（当前已禁用） |
| P2 | BUG-007 色彩错误 | - | 需新增图像色彩验证 |
| P2 | BUG-008 FPS 过低 | - | 依赖 BUG-002 修复后复测 |

### 3.3 测试增强策略

1. **BUG-001 专项测试**: 在 `test_basic_playback` 或新建 `test_playlist_switch` 中增加「播完自动切换」「播放列表双击切换」场景，结合进程输出监控
2. **BUG-003 专项测试**: 在 `test_av_sync` 的「Seek 后同步」中，增加「Seek 后音频可听」的验证（若 UI 可检测）
3. **BUG-005 专项测试**: 在 `test_edge_cases` 或新建场景中，高码率/快速 Seek 时验证非黑屏

### 3.4 数据流与错误处理

- **日志**: 修复时在关键路径增加 spdlog 日志，便于 systematic-debugging 的 Phase 1
- **锁顺序**: 遵循 AGENTS.md：先 VideoThread，后 AudioThread
- **资源**: 使用 RAII，确保 stop/reset 时资源正确释放

### 3.5 风险与应对

| 风险 | 应对 |
|------|------|
| 测试环境不稳定 | 增加重试、延长等待、记录截图 |
| 根因难以定位 | 使用 systematic-debugging，3 次失败后质疑架构 |
| 修复引入新 BUG | 每次修复后运行全量测试 |
