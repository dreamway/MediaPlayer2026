# WZMediaPlayer 小步快跑式重构计划

## 重构目标

参考 QMPlayer2 的成熟架构，通过小步快跑的方式逐步重构 WZMediaPlayer，解决以下核心问题：

1. **FFmpegView、Movie 和 MainWindow 的关系不够清晰，耦合过重**
2. **videoDecoder 目录内的代码不够清晰，特别是缓存管理及 Seeking 部分（经常会崩溃）**
3. **MainWindow 中的调用，特别是视频切换**

## 重构原则

1. **小步快跑**：每个阶段都是独立的、可验证的小改动
2. **保持功能**：每个阶段完成后，功能必须完全正常
3. **渐进式**：不一次性大改，而是逐步改进
4. **可回滚**：每个阶段都可以独立回滚
5. **测试驱动**：每个阶段完成后进行功能验证和回归测试

## 当前架构问题分析

### 问题1：FFmpegView、Movie、MainWindow 耦合过重

**现状**：
- `FFmpegView` 通过友元直接访问 `Movie::currentFrame()`
- `MainWindow` 直接操作 `Movie` 和 `FFmpegView`
- 三者之间状态同步复杂，容易出现不一致

**影响**：
- 难以单独测试某个组件
- 修改一个组件可能影响其他组件
- 状态管理混乱

### 问题2：videoDecoder 目录代码不够清晰

**现状**：
- `Video` 和 `Audio` 类职责混乱（解码 + 队列管理 + 线程管理）
- Seeking 逻辑分散在 `Movie::startDemux()`、`Video::start()`、`Audio::start()` 中
- 缓存管理复杂，容易出现竞态条件和崩溃

**影响**：
- Seeking 操作经常崩溃
- 缓存状态难以追踪
- 线程同步问题频发

### 问题3：MainWindow 视频切换复杂

**现状**：
- 需要手动调用 `Stop()`、等待线程退出、清理资源、重新 `Open()`
- 状态检查复杂，容易出现竞态条件
- 切换过程中容易出现 UI 卡顿

**影响**：
- 视频切换不稳定
- 代码重复且容易出错

## 阶段2.5：重构为 QMPlayer2 风格的架构（重要调整）

**目标**：参考 QMPlayer2 架构，将 Movie 重构为清晰的线程分离架构

**预计时间**：4-5天

**背景和反思**：
- **原计划问题**：简单地将 Movie 的功能合并到 PlayController，只是换了个名字，没有改变本质架构
- **QMPlayer2 架构优势**：
  - PlayClass：主控制器，只负责协调线程、管理状态、处理用户命令
  - DemuxerThr：独立的解复用线程类，负责读取数据包
  - VideoThr：独立的视频线程类，负责视频解码和同步
  - AudioThr：独立的音频线程类，负责音频解码和播放
  - 职责清晰，易于维护和测试
- **当前架构问题**：
  - Movie 包含所有功能（解复用、解码、线程管理混在一起）
  - 线程管理逻辑分散在 Movie::startDemux()、Video::start()、Audio::start() 中
  - 难以单独测试和维护

**重构方向**：
- PlayController 应该像 PlayClass，只负责协调和管理
- 创建独立的线程类：DemuxerThread、VideoThread、AudioThread
- 每个线程类独立管理自己的生命周期
- 通过共享的数据结构（PacketBuffer）进行线程间通信

### 2.5.1 分析当前架构并制定重构计划（1天）

**任务**：
- 分析 Movie 的所有功能
- 分析 QMPlayer2 的架构设计
- 确定重构策略：如何将 Movie 的功能拆分到独立的线程类中
- 制定详细的重构计划

**当前 Movie 的功能分析**：
1. **解复用逻辑**：`startDemux()` - 应该移到 `DemuxerThread`
2. **视频解码逻辑**：`Video::start()` - 应该移到 `VideoThread`
3. **音频解码逻辑**：`Audio::start()` - 应该移到 `AudioThread`
4. **播放控制**：PlayPause, Stop, SetSeek - 保留在 PlayController
5. **状态管理**：PlayState, IsPlaying, IsPaused, IsStopped - 保留在 PlayController
6. **解码器初始化**：streamComponentOpen, initHWDecoder - 可以保留在 PlayController 或移到线程类
7. **时钟同步**：clockbase_, sync_, getMasterClock, getClock - 保留在 PlayController
8. **音量控制**：SetVolume, GetVolume, ToggleMute - 保留在 PlayController
9. **流信息**：mVideoStreamIndex, mAudioStreamIndex, totalMilliseconds 等 - 保留在 PlayController

**QMPlayer2 架构的关键点**：
- **PlayClass**：只负责协调，不直接执行解码逻辑
- **独立线程类**：每个线程类继承 QThread，有自己的 run() 方法
- **共享数据结构**：PacketBuffer 用于线程间通信
- **清晰的职责分离**：解复用、解码、渲染各司其职

**重构策略**：
1. **PlayController**（类似 PlayClass）：
   - 持有 DemuxerThread、VideoThread、AudioThread 的指针
   - 管理播放状态（PlayState）
   - 处理用户命令（play/pause/stop/seek）
   - 管理共享数据结构（PacketBuffer）
   - 提供音视频同步相关的变量和方法

2. **DemuxerThread**（新创建）：
   - 继承 QThread 或 std::thread
   - 持有 Demuxer 实例
   - run() 方法中循环读取数据包
   - 将数据包放入 PacketBuffer（vPackets, aPackets）

3. **VideoThread**（重构 Video）：
   - 将 Video::start() 的逻辑移到 VideoThread::run()
   - 从 PacketBuffer 获取数据包
   - 解码后放入帧缓冲区
   - 处理音视频同步

4. **AudioThread**（重构 Audio）：
   - 将 Audio::start() 的逻辑移到 AudioThread::run()
   - 从 PacketBuffer 获取数据包
   - 解码后输出到音频设备
   - 更新音频时钟供视频同步使用

**验证点**：
- [x] 架构分析完成
- [x] 重构策略制定完成
- [x] 理解了 QMPlayer2 的架构设计

**问题分析**：

1. **黑画面问题**：

   - **原因**：帧在 FrameBuffer 移动过程中可能丢失数据，或帧格式未正确设置
   - **影响**：仅影响渲染显示，不影响解码、Seeking、切换等核心功能
   - **优先级**：中（不影响核心功能，但影响用户体验）

2. **频繁切换崩溃**：
   - **现象**：频繁切换视频时程序崩溃
   - **可能原因**：线程清理不彻底、资源释放时机不对、竞态条件

   - **优先级**：高（需要优先解决）
