# WZMediaPlayer 架构分析与稳定性改进方案

## 一、当前架构分析

### 1.1 架构概览

```
PlayController (主控制器)
├── DemuxerThread (解复用线程)
│   └── PacketQueue (vPackets_, aPackets_)
├── VideoThread (视频解码线程)
│   ├── Decoder (解码器)
│   └── VideoWriter (渲染器)
└── AudioThread (音频解码线程)
    └── Audio (音频输出)
```

### 1.2 架构优势

1. **职责分离清晰**：每个线程负责单一职责
2. **接口抽象良好**：Decoder和VideoWriter接口设计合理
3. **参考成熟方案**：基于QMPlayer2的成熟架构

### 1.3 架构问题

1. **线程同步复杂**：多线程之间的同步机制复杂，容易出现竞态条件
2. **状态管理分散**：Seeking状态、播放状态分散在多个类中
3. **错误恢复不足**：缺乏完善的错误恢复机制
4. **资源管理不统一**：资源释放时机不明确

## 二、关键问题分析

### 2.1 Seeking后视频停止渲染（高优先级）

**问题现象**：
- Seeking操作后，VideoThread停止渲染
- 日志显示VideoThread在检查videoSeekPos后没有继续解码

**根本原因**：
1. `videoSeekPos`检查逻辑问题：检查位置不当，导致无法继续解码
2. `wasSeeking`状态管理混乱：局部变量和成员变量混用
3. 线程同步问题：seeking期间线程锁定可能导致死锁

**修复方案**：
1. ✅ 已修复：将`videoSeekPos`检查移到解码之后
2. ✅ 已修复：统一`wasSeekingRecently_`状态管理，在成功解码关键帧后立即清除
3. ✅ 已修复：优化线程锁定机制，使用统一的锁顺序，避免死锁
4. ✅ 已修复：优化队列清空时机，在`requestSeek()`时立即清空
5. ✅ 已修复：实现KeyFrame直接跳转，使用`AVSEEK_FLAG_FRAME`避免"先跳回原点"的问题
6. ✅ 已修复：优化缓冲管理，在seeking期间直接丢弃满队列的数据包

### 2.2 音视频同步不精确（中优先级）

**问题现象**：
- 音视频同步机制已实现，但准确性待验证
- 同步阈值和调整策略可能需要优化

**根本原因**：
1. 同步算法简单：仅使用简单的延迟调整
2. 时钟管理分散：音频时钟和视频时钟管理分散
3. 同步阈值固定：没有根据实际情况动态调整

**修复方案**：
1. 优化同步算法：参考QMPlayer2的同步策略
2. 统一时钟管理：在PlayController中统一管理
3. 动态调整阈值：根据播放情况动态调整

### 2.3 线程安全问题（高优先级）

**问题现象**：
- 多线程访问共享资源时可能出现竞态条件
- Seeking期间可能出现死锁

**根本原因**：
1. 锁的使用不当：锁的粒度过大或过小
2. 锁的顺序问题：多个锁的获取顺序不一致
3. 条件变量使用不当：条件变量的等待和唤醒时机不当

**修复方案**：
1. 优化锁的粒度：减小锁的粒度，提高并发性
2. 统一锁的顺序：确保所有线程按相同顺序获取锁
3. 改进条件变量：使用更精确的条件变量机制

### 2.4 错误处理不足（中优先级）

**问题现象**：
- 解码错误时缺乏恢复机制
- 线程异常时无法自动恢复

**根本原因**：
1. 异常处理不完善：关键路径缺乏异常保护
2. 错误恢复机制缺失：错误发生后无法自动恢复
3. 日志记录不足：错误信息记录不完整

**修复方案**：
1. 完善异常处理：在所有关键路径添加异常保护
2. 实现错误恢复：添加自动错误恢复机制
3. 改进日志记录：记录更详细的错误信息

### 2.5 资源管理问题（中优先级）

**问题现象**：
- 可能存在资源泄漏
- 资源释放时机不明确

**根本原因**：
1. RAII使用不充分：部分资源未使用RAII管理
2. 资源释放时机不当：资源释放时机不明确
3. 异常安全不足：异常时资源可能无法正确释放

**修复方案**：
1. 全面使用RAII：所有资源使用RAII管理
2. 明确释放时机：明确资源释放的时机
3. 确保异常安全：确保异常时资源也能正确释放
