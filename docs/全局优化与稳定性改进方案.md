# WZMediaPlayer 全局优化与稳定性改进方案

## 一、问题分析

### 1.1 当前问题总结

1. **崩溃问题**（高优先级）
   - 运行过程中突然崩溃，日志截断
   - 可能原因：OpenGL上下文销毁后访问、线程删除时仍在运行、资源竞态条件

2. **跳帧闪烁**（中优先级）
   - 跳帧时直接clear()然后continue，没有平滑过渡
   - 导致画面闪烁

3. **状态管理分散**（高优先级）
   - 播放状态分散在多个类中（PlayController、VideoThread、AudioThread）
   - Seeking状态、播放状态没有统一管理
   - 状态转换缺乏验证

4. **线程同步复杂**（高优先级）
   - 锁的使用不当，可能导致死锁
   - 条件变量使用不当
   - 缺乏超时机制

5. **错误恢复不足**（中优先级）
   - 解码错误时缺乏恢复机制
   - 线程异常时无法自动恢复

6. **资源管理不统一**（中优先级）
   - 资源释放时机不明确
   - RAII使用不充分

## 二、全局优化方案

### 2.1 状态机管理（核心改进）

**目标**：统一管理播放状态，确保状态转换的正确性和原子性

**实现方案**：
1. 创建`PlaybackStateMachine`类，统一管理播放状态
2. 定义清晰的状态和转换规则
3. 状态转换时验证前置条件
4. 使用原子操作确保线程安全

**状态定义**：
```cpp
enum class PlaybackState {
    Idle,           // 空闲状态
    Opening,        // 正在打开文件
    Ready,          // 文件已打开，准备播放
    Playing,        // 正在播放
    Paused,         // 已暂停
    Seeking,        // 正在Seeking
    Stopping,       // 正在停止
    Stopped,        // 已停止
    Error           // 错误状态
};
```

**状态转换规则**：
- Idle -> Opening -> Ready -> Playing
- Playing <-> Paused
- Playing -> Seeking -> Playing
- Any -> Stopping -> Stopped
- Any -> Error

### 2.2 线程同步优化

**目标**：简化线程同步机制，避免死锁和竞态条件

**实现方案**：
1. 使用`std::shared_mutex`替代多个`std::mutex`
2. 统一锁的获取顺序
3. 所有锁操作添加超时机制
4. 使用`std::atomic`替代需要同步的简单标志

**锁顺序规则**：
1. PlayController锁（如果存在）
2. VideoThread锁
3. AudioThread锁
4. DemuxerThread锁

### 2.3 错误处理和恢复机制

**目标**：实现统一的错误处理和自动恢复机制

**实现方案**：
1. 创建`ErrorRecoveryManager`类
2. 定义错误类型和恢复策略
3. 自动重试机制
4. 错误统计和报告

**错误类型**：
- 解码错误：自动flush解码器并重试
- 网络错误：重试3次后放弃
- 资源错误：释放资源并报告
- 线程错误：重启线程

### 2.4 跳帧闪烁优化

**目标**：减少跳帧时的闪烁

**实现方案**：
1. 跳帧时不立即clear()，而是标记为"待跳过"
2. 在渲染前检查标记，如果标记为跳过，使用上一帧
3. 或者使用双缓冲，跳帧时切换到备用缓冲

### 2.5 资源管理统一

**目标**：统一资源管理，确保异常安全

**实现方案**：
1. 所有资源使用RAII管理
2. 使用`std::unique_ptr`和`std::shared_ptr`
3. 明确资源生命周期
4. 确保异常安全

## 四、单元测试计划

### 4.1 PlaybackStateMachine测试

```cpp
TEST(PlaybackStateMachine, StateTransitions) {
    PlaybackStateMachine sm;
    
    // 测试正常状态转换
    EXPECT_TRUE(sm.transitionTo(PlaybackState::Opening));
    EXPECT_EQ(sm.getState(), PlaybackState::Opening);
    
    EXPECT_TRUE(sm.transitionTo(PlaybackState::Ready));
    EXPECT_EQ(sm.getState(), PlaybackState::Ready);
    
    // 测试非法状态转换
    EXPECT_FALSE(sm.transitionTo(PlaybackState::Seeking));  // 不能从Ready直接到Seeking
}

TEST(PlaybackStateMachine, ConcurrentAccess) {
    PlaybackStateMachine sm;
    
    // 测试并发访问
    std::vector<std::thread> threads;
    for (int i = 0; i < 10; ++i) {
        threads.emplace_back([&sm]() {
            for (int j = 0; j < 100; ++j) {
                sm.getState();
            }
        });
    }
    
    for (auto& t : threads) {
        t.join();
    }
    
    // 应该没有崩溃或数据竞争
}
```

### 4.2 ThreadSyncManager测试

```cpp
TEST(ThreadSyncManager, LockOrder) {
    ThreadSyncManager sync;
    
    // 测试锁顺序
    auto lock1 = sync.lockVideoThread();
    auto lock2 = sync.lockAudioThread();
    
    // 应该能正常获取锁
    EXPECT_TRUE(lock1.owns_lock());
    EXPECT_TRUE(lock2.owns_lock());
}

TEST(ThreadSyncManager, Timeout) {
    ThreadSyncManager sync;
    
    // 测试超时机制
    auto lock1 = sync.lockVideoThread();
    auto lock2 = sync.lockVideoThread(std::chrono::milliseconds(100));
    
    // 第二个锁应该超时
    EXPECT_FALSE(lock2.owns_lock());
}
```

### 4.3 ErrorRecoveryManager测试

```cpp
TEST(ErrorRecoveryManager, DecodeErrorRecovery) {
    ErrorRecoveryManager recovery;
    
    // 模拟解码错误
    ErrorInfo error{ErrorType::DecodeError, "Decoder returned EOF"};
    
    // 应该自动恢复
    auto result = recovery.handleError(error);
    EXPECT_EQ(result.action, RecoveryAction::FlushAndRetry);
    EXPECT_EQ(result.retryCount, 1);
}
```
